# Next Academy — i18n (Internationalization)

> Last Updated: 2026-03-13 04:00

---

## 1. Supported Languages

1. **Arabic (ar)** — RTL (Right-to-Left) — Default
2. **English (en)** — LTR (Left-to-Right)

---

## 2. Implementation Strategy

- **Library:** `next-intl` (Next.js App Router compatible)
- **URL structure:** `/ar/programs` and `/en/programs`
- **Default locale:** `ar` (Arabic is default, no `/ar` prefix needed optionally)
- **Detection:** Browser `Accept-Language` → cookie → default `ar`
- **User preference:** Saved in `user.preferredLanguage` field
- **Language toggle:** Visible in Navbar on all pages

### 2.1 File Structure

```text
messages/
├── ar.json → Arabic translations (ALL UI strings)
├── en.json → English translations
└── Structure:
    ├── common: shared strings (buttons, labels, nav)
    ├── auth: login, register, forgot password
    ├── booking: booking flow strings
    ├── checkout: payment and checkout strings
    ├── dashboard: user dashboard strings
    ├── instructor: instructor portal strings
    ├── programs: program listing and details
    ├── reviews: review system strings
    ├── errors: error messages
    ├── notifications: notification messages
    └── onboarding: onboarding flow strings
```

### 2.2 Translation Rules

```text
Rules:
├── NEVER hardcode text in components — always use t('key')
├── Every new string → add to BOTH ar.json and en.json
├── Keys use dot notation: t('booking.confirmButton')
├── Pluralization: use ICU MessageFormat
│   Example: "{count, plural, =0 {لا توجد حجوزات} one {حجز واحد} two {حجزان} few {# حجوزات} many {# حجزاً} other {# حجز}}"
├── Variables: use named interpolation {name}, {amount}, {date}
│   Example: t('booking.greeting', { name: user.firstName })
├── HTML in translations: use rich text format
│   Example: t.rich('terms.agreement', { link: (chunks) => <a>{chunks}</a> })
└── Missing key fallback: show key name in development, fallback to AR in production
```

## 3. Translation Scope

| Content Type | Translation Method |
|---|---|
| UI Strings (buttons, labels, nav) | JSON translation files (`messages/ar.json`, `messages/en.json`) |
| Program titles & descriptions | Admin enters both versions in Payload CMS (dual fields: `title_ar`, `title_en`) |
| Instructor bios | Admin enters both versions (dual fields: `bio_ar`, `bio_en`) |
| Category names | Dual fields in Payload CMS |
| Blog posts | Admin writes in chosen language, optional translation field |
| Emails & WhatsApp templates | Separate templates per language |
| Legal pages | Separate markdown files per language: `privacy-ar.md`, `privacy-en.md` |
| Certificates | Template per language (determined by user's locale at completion) |
| PDF Invoices | Generated in user's preferred language |
| Error messages | Dual messages in API responses (`message` + `message_en`) |
| Push notifications | Sent in user's `preferredLanguage` |
| Admin panel | English only (Payload CMS default) |

## 4. RTL Considerations

### 4.1 CSS Layout

```text
Global:
├── CSS dir: rtl applied when locale === 'ar'
├── Body: direction and font-family changed per locale
├── Flexbox: no row-reverse needed if using logical properties ✅
└── Grid: automatically mirrors in RTL

Logical Properties (MUST USE):
├── margin-inline-start → instead of margin-left/right
├── padding-inline-end → instead of padding-right/left
├── inset-inline-start → instead of left/right
├── border-inline-end → instead of border-right/left
├── text-align: start → instead of text-align: left
└── float: inline-start → instead of float: left
```

### 4.2 Component-Specific RTL

```text
Must Flip:
├── Arrow icons (→ becomes ←)
├── Chevrons in breadcrumbs and menus
├── Progress bars (fill from right in RTL)
├── Carousel arrows (next = left, prev = right in RTL)
├── Checkmarks in checkboxes (mirror)
├── Sidebar position (right side in RTL)
├── Dropdown menu alignment
└── Form label position (right of input in RTL)

Must NOT Flip:
├── Phone numbers (always LTR)
├── Prices and numeric values (always LTR digits)
├── URLs and email addresses
├── Code snippets
├── Brand logos
├── Video player controls
├── Clock time displays
└── Credit card number input
```

### 4.3 Font System

```text
Arabic (RTL):
├── Primary: Cairo (variable weight: 300-900)
├── Fallback: system-ui, -apple-system, sans-serif
├── Body: Cairo 400 (Regular)
├── Headings: Cairo 700 (Bold)
└── Brand: Cinzel (Latin only, for logo/brand text)

English (LTR):
├── Primary: Montserrat (headings)
├── Secondary: Inter (body text)
├── Fallback: system-ui, -apple-system, sans-serif
├── Body: Inter 400
├── Headings: Montserrat 700
└── Brand: Cinzel

Font Loading:
├── next/font for automatic optimization
├── font-display: swap (prevent invisible text)
├── Subset: Arabic + Latin scripts only
└── Preload: Cairo for AR, Inter for EN (above-fold only)
```

## 5. Date & Number Formatting

### 5.1 Date Formats

| Locale | Short Date | Long Date | Time | Example |
|---|---|---|---|---|
| **ar** | `DD/MM/YYYY` | `DD MMMM YYYY` (هجري ممكن) | `HH:mm` | ١٥ مارس ٢٠٢٦ - ٠٣:٣٠ |
| **en** | `MM/DD/YYYY` | `MMMM DD, YYYY` | `hh:mm A` | March 15, 2026 - 3:30 PM |

```typescript
// Use Intl.DateTimeFormat
const formattedDate = new Intl.DateTimeFormat(locale, {
  year: 'numeric',
  month: 'long',
  day: 'numeric',
}).format(date);

// Relative time: "منذ ساعتين" / "2 hours ago"
const relativeTime = new Intl.RelativTimeFormat(locale).format(-2, 'hours');
```

### 5.2 Number Formats

```text
Arabic numerals:
├── Option 1: Eastern Arabic numerals (٠١٢٣٤٥٦٧٨٩) — more authentic
├── Option 2: Western Arabic numerals (0123456789) — more universally readable
├── Decision: Use Western Arabic (0-9) for all numbers (user familiarity)
├── Thousands separator: comma (3,000)
└── Decimal separator: period (3,000.50)
```

### 5.3 Currency Display

| Locale | Format | Example |
|---|---|---|
| **ar** | `[amount] ج.م` | ٣٬٠٠٠ ج.م |
| **en** | `EGP [amount]` | EGP 3,000 |

```typescript
const formattedPrice = new Intl.NumberFormat(locale, {
  style: 'currency',
  currency: 'EGP',
  currencyDisplay: locale === 'ar' ? 'narrowSymbol' : 'code',
}).format(amount);
```

## 7. User Language Switch

```text
Flow:
1. User clicks language toggle (AR ↔ EN) in Navbar
2. Current page re-renders in new locale
3. URL changes: /ar/programs → /en/programs
4. If logged in: user.preferredLanguage updated in DB
5. If guest: saved to cookie (NEXT_LOCALE) + localStorage
6. All future notifications/emails sent in preferred language
7. Page content (CMS): switches to _en fields if available, fallback to _ar

Edge Cases:
├── CMS content missing English version → show Arabic as fallback
├── Translation key missing → show key in dev, Arabic fallback in prod
├── User switches mid-checkout → preserve all form data
├── User switches mid-onboarding → preserve step progress
└── Deep-linked URL with locale → respect URL locale over preference
```
